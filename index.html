Import Necessary Libraries for PyTorch
Handling Tensors and Core PyTorch Operations

Core Concept

PyTorch's ecosystem is organized into torch (core), torch.nn (neural network layers), torch.optim (optimizers), torch.utils.data (data utilities), and torchvision (vision utilities). Knowing exactly what each submodule provides is essential.

Essential Imports Reference

import torch.nn as nn               # Layers, loss functions
import torch.nn.functional as F      # Stateless functions (relu, softmax…)
import torch.optim as optim          # SGD, Adam, etc.
from torch.utils.data import Dataset, DataLoader
import torchvision
import torchvision.transforms as transforms
import numpy as np

# Quick sanity check
print(torch.__version__)              # e.g. 2.2.0
print(torch.cuda.is_available())      # True / False
torch Core tensors, autograd, device management
torch.nn Stateful layers (Linear, Conv2d, LSTM…)
torch.nn.functional Stateless ops (F.relu, F.cross_entropy)
torch.optim Optimizers with state (momentum, lr)
● MCQ
Which module should you import to use F.relu() and F.cross_entropy() as stateless functional operations?
A torch.nn
B torch.nn.functional
C torch.optim
D torch.autograd
✓ Answer: B

torch.nn.functional (typically aliased as F) contains stateless versions of activation functions, loss functions, and other ops. Unlike nn.ReLU() (a module with state), F.relu() is a pure function. Both produce the same output, but F.* is commonly used inside forward() methods.

◆ Objective
True or False: torch.nn and torch.nn.functional always produce identical outputs for the same operation (e.g., ReLU), and they are completely interchangeable.
✓ Answer: False

They produce the same numerical output, but they are NOT always interchangeable. nn.Dropout(p=0.5) respects model.train()/model.eval() mode, while F.dropout(x, p=0.5) requires you to pass training=model.training explicitly. Forgetting this is a common bug. Also, nn.* modules are registered parameters/submodules visible to the optimizer; F.* calls are not.

PyTorch Fundamentals · Working with PyTorch Tensor
Convert NumPy Arrays to PyTorch Tensors

Core Concept

Two main conversion paths exist: torch.from_numpy() (shares memory — zero-copy) and torch.tensor() (always copies). Memory sharing means mutating one changes the other. This is a frequent exam trap.

import numpy as np
import torch

arr = np.array([1.0, 2.0, 3.0])

# Method 1: SHARES memory with arr
t1 = torch.from_numpy(arr)
arr[0] = 99.0
print(t1)   # tensor([99.,  2.,  3.], dtype=torch.float64) ← CHANGED!

# Method 2: COPIES (type inference)
t2 = torch.tensor(arr)
arr[0] = 0.0
print(t2)   # tensor([99.,  2.,  3.], dtype=torch.float64) ← unchanged

# Convert back: .numpy() — also shares memory
t3 = torch.tensor([4.0, 5.0])
np_back = t3.numpy()   # Only works on CPU tensors

# dtype conversion during conversion
t4 = torch.tensor(arr, dtype=torch.float32)
from_numpy() only works with CPU tensors. If your tensor is on GPU, call .cpu() first before .numpy(). Also, from_numpy() does NOT support all NumPy dtypes — e.g., complex128 arrays may fail.
⌨ Code Output
What does the following code print? 
import numpy as np, torch
a = np.array([10, 20, 30])
t = torch.from_numpy(a)
t[1] = 999
print(a)
A [10, 20, 30] — tensor and array are separate
B [10, 999, 30] — shared memory
C RuntimeError: tensors share storage
D [999, 999, 999]
✓ Answer: B

torch.from_numpy() creates a tensor that shares underlying memory with the NumPy array. Modifying the tensor element at index 1 also changes a[1]. This zero-copy bridge is powerful but requires care — always use torch.tensor(arr) if you want an independent copy.

Initialize Tensors with Random and Constant Values

Core Concept

PyTorch provides a rich set of tensor factory functions. Know the difference between creating tensors from scratch vs. creating tensors with the same shape/dtype as an existing tensor (the _like family).

# --- Constant tensors ---
torch.zeros(3, 4)            # all zeros, shape [3,4]
torch.ones(2, 3)             # all ones
torch.full((3, 3), 7.0)       # fill with 7.0
torch.eye(4)                  # 4x4 identity matrix
torch.arange(0, 10, 2)       # [0, 2, 4, 6, 8]
torch.linspace(0, 1, 5)      # [0.0, 0.25, 0.5, 0.75, 1.0]

# --- Random tensors ---
torch.rand(3, 4)              # Uniform [0, 1)
torch.randn(3, 4)             # Standard Normal N(0,1)
torch.randint(0, 10, (3, 4))  # Random ints in [0, 10)
torch.randperm(5)             # Random permutation of [0..4]

# --- _like family (same shape/dtype as existing tensor) ---
x = torch.tensor([[1.0, 2.0], [3.0, 4.0]])
torch.zeros_like(x)           # zeros, shape [2,2], float32
torch.ones_like(x)
torch.rand_like(x)            # uniform random, same shape/dtype

# --- Reproducibility ---
torch.manual_seed(42)
torch.rand(2, 2)              # deterministic with the seed

● MCQ
You need a tensor of shape (4, 4) filled with random integers between 5 (inclusive) and 15 (exclusive). Which call is correct?
A torch.rand(4, 4) * 10 + 5
B torch.randn(4, 4, low=5, high=15)
C torch.randint(5, 15, (4, 4))
D torch.arange(5, 15).reshape(4, 4) (only works if there are exactly 10 elements)
✓ Answer: C

torch.randint(low, high, size) is the canonical way to get random integers in [low, high). Note: size must be a tuple, not individual ints like torch.rand accepts. Option A produces floats, not integers. Option D would fail because arange(5,15) has 10 elements and cannot reshape to (4,4)=16 elements.

◆ Objective
What is the difference between torch.rand_like(x) and torch.rand(x.shape)?
✓ Answer

torch.rand_like(x) creates a tensor with the same shape, dtype, device, and layout as x. torch.rand(x.shape) always creates a CPU float32 tensor regardless of where x lives. If x is on GPU or has dtype float16, rand_like automatically matches it — making it the preferred choice in model code.

● MCQ
You have a model and input tensors on CPU. The following code raises a RuntimeError. Why? 
model = MyModel().cuda()
x = torch.rand(1, 10)

A MyModel cannot be used with CUDA

◆ Objective
A tensor was saved on a GPU machine. Your deployment environment only has CPU. How do you load it without errors?

Use the map_location parameter: torch.load('tensor.pt', map_location=torch.device('cpu'), weights_only=True). The map_location argument remaps device locations during deserialization. You can also use a string: map_location='cpu'. This is essential when deploying GPU-trained models to CPU inference servers.

● MCQ
What is the difference between torch.set_num_threads() and torch.set_num_interop_threads()?
A set_num_threads controls parallelism within a single op; set_num_interop_threads controls parallelism between independent ops
B set_num_threads controls GPU threads; set_num_interop_threads controls CPU threads
C They are aliases for the same function
D set_num_interop_threads only affects DataLoader workers
✓ Answer: A

Intra-op parallelism (set_num_threads) splits a single operation like matmul across multiple threads. Inter-op parallelism (set_num_interop_threads) runs independent operations concurrently. Both must be set before any computation starts — they cannot be changed mid-run. For most single-machine use cases, set_num_threads is the one to tune.

a = x[idx]   # copy — modifying a won't affect x
b = x[0:2]   # view — shares memory with x

Split and Reshape Tensors
x = torch.arange(12).reshape(4, 3)

# split: divide into chunks of given size along dim
chunks = torch.split(x, 2, dim=0)  # two tensors of shape [2,3]
chunks = torch.split(x, [1,3], dim=0)  # [1,3] and [3,3] shapes

# chunk: divide into N equal chunks
parts = torch.chunk(x, 2, dim=0)   # two [2,3] tensors
# Note: if not evenly divisible, last chunk is smaller

# reshape vs view
y = x.reshape(2, 6)    # always works; may copy
y = x.view(2, 6)       # zero-copy; fails if non-contiguous
x.contiguous().view(2, 6)  # safe pattern

# Transpose makes tensor non-contiguous!
t = x.T               # or x.transpose(0, 1)
t.is_contiguous()     # False
t.contiguous().view(-1)  # now works
view() raises RuntimeError if the tensor is not contiguous in memory (e.g., after transpose). Always use .contiguous().view() or just use .reshape() which handles non-contiguous tensors automatically.
● MCQ
You have a tensor x of shape [6, 4] and run x.T.view(-1). What happens?
A Returns a 1D tensor of 24 elements
B Raises RuntimeError: tensor is not contiguous
C Returns a 1D tensor but in column-major order
D Returns a tensor of shape [4, 6]
✓ Answer: B

x.T (transpose) returns a non-contiguous view of x. Calling .view() on a non-contiguous tensor raises RuntimeError. Fix: x.T.contiguous().view(-1) or x.T.reshape(-1). This is one of the most common PyTorch bugs in production code.

Implement Tensor Views
x = torch.arange(24)  # shape [24]

# view: zero-copy reshape (requires contiguous memory)
y = x.view(4, 6)
y.data_ptr() == x.data_ptr()   # True — same memory
y[0, 0] = 999
x[0]                         # 999 — both changed!

# expand: broadcast a size-1 dim without copying
z = torch.zeros(3, 1)
z_exp = z.expand(3, 4)    # shape [3, 4] — ZERO COPY
z_exp.is_contiguous()      # False (stride is 0 for expanded dim)

# repeat: actually copies data
z_rep = z.repeat(1, 4)     # shape [3, 4] — COPIES

# as_strided: low-level view with custom strides
x2 = torch.arange(9)
# view as 3x3 with stride [3, 1]
y2 = torch.as_strided(x2, size=(3, 3), stride=(3, 1))
◆ Objective
What is the key difference between expand() and repeat()? When would you choose one over the other?
✓ Answer

expand() creates a view with zero-copy by setting the stride of the expanded dimension to 0. It is memory-efficient but the result is non-contiguous and read-only in the expanded dimension. repeat() physically copies data, creating a full, contiguous tensor. Choose expand() when you only need to read the expanded tensor (e.g., broadcasting in a loss function). Use repeat() when you need a writable, contiguous copy or will modify the result.

● MCQ
What does pin_memory=True in DataLoader do?
A Pins the model weights in RAM to prevent paging
B Allocates batches in pinned (page-locked) CPU memory, enabling faster async GPU transfer

Normalize Data for Model Training
import torchvision.transforms as transforms

# For image data (e.g., ImageNet normalization)
transform = transforms.Compose([
    transforms.ToTensor(),              # PIL → [0,1] float tensor
    transforms.Normalize(
        mean=[0.485, 0.456, 0.406],    # per-channel mean
        std=[0.229, 0.224, 0.225]       # per-channel std
    )
])

# Manual normalization of tabular data
X = torch.randn(100, 10)   # raw features
mean = X.mean(dim=0)       # shape [10]
std  = X.std(dim=0)        # shape [10]
X_norm = (X - mean) / (std + 1e-8)   # small epsilon for stability

# IMPORTANT: compute stats on train set only, apply to val/test!
X_val_norm = (X_val - mean) / (std + 1e-8)   # use train mean/std!
Always compute normalization statistics (mean, std) on the TRAINING set only. Applying training statistics to val/test is correct. Computing stats on the full dataset causes data leakage.
◆ Objective
What does transforms.ToTensor() do to a PIL image, and why is it usually placed before Normalize()?
✓ Answer

ToTensor() converts a PIL Image (or NumPy HxWxC uint8 array) to a PyTorch float tensor of shape [C, H, W] with values in [0.0, 1.0] (divides by 255). It must come before Normalize() because Normalize expects a float tensor with values in [0, 1] — it applies (x - mean) / std per channel. Applying Normalize to uint8 [0, 255] pixel values with ImageNet statistics would produce meaningless results.

Develop Neural Network Models Using Custom Layers
import torch.nn as nn
import torch

# Custom layer: subclass nn.Module
class LinearWithBias(nn.Module):
    def __init__(self, in_features, out_features):
        super().__init__()                         # MUST call super()
        self.weight = nn.Parameter(              # Learnable parameter
            torch.randn(out_features, in_features) * 0.01
        )
        self.bias = nn.Parameter(torch.zeros(out_features))

    def forward(self, x):
        return x @ self.weight.T + self.bias   # [B, in] @ [in, out] + [out]

# Full custom model
class MyMLP(nn.Module):
    def __init__(self, in_dim, hidden, out_dim):
        super().__init__()
        self.fc1 = nn.Linear(in_dim, hidden)
        self.fc2 = nn.Linear(hidden, out_dim)

    def forward(self, x):
        x = torch.nn.functional.relu(self.bn(self.fc1(x)))
        return self.fc2(x)

● MCQ
Why is optimizer.zero_grad() called at the beginning of each training step?
A To reset the model's weights to zero before each forward pass
B Because PyTorch accumulates gradients by default — not clearing them would sum gradients from multiple batches, corrupting the update
C To free GPU memory used by the computation graph
D To synchronize gradients across distributed workers
✓ Answer: B

PyTorch accumulates (adds) gradients in .grad tensors by default. This is intentional — it enables gradient accumulation as a technique for effective large batch sizes. However, in a standard training loop you must zero them each step. Forgetting zero_grad() is one of the most common PyTorch bugs — loss may initially decrease but training becomes erratic as gradients from previous batches distort the update direction. Alternative: optimizer.zero_grad(set_to_none=True) is slightly faster (sets grad to None instead of zeroing).

B nn.BCEWithLogitsLoss()
C nn.MSELoss()

Monitor and Interpret Model Performance Metrics During Training
# Typical training loop with metric tracking
for epoch in range(num_epochs):
    model.train()
    train_loss = 0.0
    correct = 0

    for X, y in train_loader:
        X, y = X.to(device), y.to(device)
        optimizer.zero_grad()
        out = model(X)
        loss = criterion(out, y)
        loss.backward()
        optimizer.step()

        train_loss += loss.item()          # .item() detaches from graph
        preds = out.argmax(dim=1)
        correct += (preds == y).sum().item()

    avg_loss = train_loss / len(train_loader)
    accuracy = correct / len(train_loader.dataset)
    print(f"Epoch {epoch}: loss={avg_loss:.4f}, acc={accuracy:.4f}")

    # Validation
    model.eval()
    with torch.no_grad():                    # disable gradient tracking
        # ... compute val metrics
        pass
Always call loss.item() to extract scalar loss value. Using loss directly accumulates computation graphs in memory, eventually causing OOM (Out-of-Memory) errors during long training runs.
◆ Objective
Why is torch.no_grad() used during validation, and what happens if you omit it?
✓ Answer

torch.no_grad() disables gradient computation — no computational graph is built for operations within its context. During validation: (1) Memory usage is dramatically lower because intermediate tensors don't need to be retained for backprop. (2) Computation is faster (~30–50% speedup). If omitted, PyTorch builds the full computation graph for every validation batch, eventually exhausting GPU/CPU memory during long validation loops. It does NOT affect forward pass outputs — results are numerically identical. Use it as a context manager or decorator for any inference code.

dist.init_process_group(

✎ Subjective
Your model produces NaN loss after epoch 3 during training. Describe a systematic debugging approach using PyTorch's built-in tools to diagnose the root cause.
✓ Reference Answer

Step 1 — Enable Anomaly Detection: Wrap the forward/backward pass with torch.autograd.detect_anomaly(). This will pinpoint the exact operation that first produces NaN with a full stack trace.

Step 2 — Check Gradients: After loss.backward(), iterate named_parameters and check for NaN grads using torch.isnan(param.grad).any(). Exploding gradients often precede NaN — use gradient clipping (clip_grad_norm_).

Step 3 — Register Forward Hooks: Add hooks to each layer to inspect intermediate activations. NaN often originates from log(0) in a loss function, division by zero in normalization, or overflow in float16 operations.

Step 4 — Check Input Data: Verify training data contains no NaN/Inf values using torch.isnan(X).any(). A corrupted batch can propagate NaN through the entire graph.

Step 5 — Learning Rate: An excessively large LR causes weight explosion. Reduce LR by 10x and monitor if NaN disappears. If using AMP (float16), check if the GradScaler's scale factor is overflowing.
